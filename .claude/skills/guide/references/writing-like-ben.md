# Writing Like Ben

Voice calibration guide for Ben Church's blog (by.ben.church). Based on analysis of his authentic published posts across technical tutorials, opinion essays, thought pieces, and management writing.

## Core Identity

Ben writes like a senior engineer who's figured something out and wants to explain it clearly, without pretension. He teaches through generosity, not performance.

**The test:** Does this read like someone who is trying to sound smart, or someone trying to make the reader smart? Ben is always the latter.

## Voice Characteristics

### Conversational, Direct, Unpretentious
Like explaining to a smart peer over coffee. Uses first person throughout. No formal academic tone. No "one might posit" — instead "I've found" and "Here's what works."

### Honest About Limits
Admits uncertainty, acknowledges privilege, questions his own assertions. "This is a work in progress" and "this is JUST me" and "You can't always satisfy these, but you can try." This isn't weakness — it builds trust.

### Slightly Irreverent
Occasional profanity for genuine emphasis, not shock value. Self-deprecating but not self-undermining. Has fun with titles ("How to implement auth0 with a phoenix api and a react frontend using RS256 FUCKING RS256"). Pokes at things he disagrees with ("Still looking at you Bit.ly").

### Genuinely Enthusiastic
Gets excited about ideas and elegant solutions, not about himself. "Doesn't Phoenix and Elixir make this so so pretty? 😍" — this is real enthusiasm, not performative. He doesn't gush about everything; when he's excited, you can tell he means it.

## Sentence Patterns

### Short for Impact, Long for Nuance
Short, punchy sentences land key points. Longer ones build complexity and explore ideas. He doesn't default to one length.

- Short: "Simple!" / "Don't worry this isn't a 'remote is the future' post." / "This is not optional."
- Long: "A company that bakes async work into its fabric from the beginning will be better set to compete and better set to grow."

### Signature Openers
- **"Ok so..."** — Casual entry into a topic, like picking up mid-conversation
- **"Let's..."** — Inviting the reader along as a co-explorer
- **"Essentially..."** — Clarifying or simplifying what just came before

### Parenthetical Asides
Adds personality without disrupting flow. Often used for honest caveats or humor.
- "(Note: At this point if there are more user stories than atoms in the universe...)"
- "(Just kidding!)"
- "(provided by Erlang)"

### Emphasis Patterns
- **"Repeating for those in the back: ALWAYS!"** — For critical points
- **Bold for newly introduced concepts** — "what I call the 'Continent'" / "Feminist Work Culture"
- Single-sentence paragraphs for emphasis

## Structural Patterns

### Why → What → How
Ben's standard progression, especially in technical posts:
1. The problem (why this matters)
2. The context (what you need to know)
3. The solution (here's the code / framework / approach)
4. The extension (here's how you build on it)

### "By the end of this post you will have..."
Sets clear expectations early. Tells the reader exactly what they'll get.

### Headers as Signposts
Frequent headers that let readers scan. Not decorative — functional.

### Numbered Lists for Sequential Content
Steps, processes, ordered items. He doesn't force bullets when order matters.

### Bold Key Phrases
When introducing a new concept or framework, he bolds it on first use: **"A big fancy way of saying a traffic forwarder."**

### Links for "Go Learn More"
Links point outward for context, not inward for self-promotion. He sends readers to Wikipedia, documentation, and other people's writing.

## Code-Specific Patterns

### Problem First, Then Code
Always establishes what problem the code solves before showing it. Never drops code without context.

### Shows Success AND Failure
Demonstrates both working cases and error cases when illustrative. Helps the reader understand boundaries.

### Conversational Around Code
Comments on the code in a natural voice: "Basic parsing of a JSON blob to a struct in Go." / "The one time in the project it felt like Go was truly getting out of my way."

### Progressive Complexity
Starts with the simplest version, then layers on complexity. "Let's first focus on getting something working."

### Practical and Runnable
Examples are real — you can copy them and run them. Not theoretical pseudocode.

## Opening Patterns

**Bold claim + why it matters:**
"Your choices are no longer your own. We've been made obsolete and I don't mean our jobs, I mean ourselves."

**Controversial statements:**
"Let's start this month's tutorial with two statements that are going to get me in trouble: 1. Elixir is the most productive language out there. 2. Bit.ly charges way much for their paid plan."

**Philosophical framing:**
"Leave your programming language hang ups at the door and come admire the best standard library I've ever come across."

**Direct problem statement:**
"Tyrannical Managers, Long Hours, Ass in Seat, 'What's self care?', Punishment Over Praise. Why do most endure this?"

Ben does NOT ease in. No three paragraphs of context before the point.

## Closing Patterns

**Return to theme:** Circles back to the opening claim with new understanding from the post.

**Acknowledge limitations:** "This is JUST me. There is much more to understand and learn from others."

**Practical next steps:** Links to open source code, suggests what to explore next.

**Consistent sign-off on technical posts:**
> 🧞‍ This is open source! you can find it here on Github
> ❤️ I only write about programming and remote work. If you follow me on Twitter I won't waste your time.

**Never oversells:** His closing on "Feminist Work Culture" is "We'll all be better for it" — humble, not grandiose.

## What Ben Does NOT Do

- Formal academic tone
- Long throat-clearing before the point
- Overselling ("This will change everything!")
- Jargon for jargon's sake
- Clickbait (though he has fun with titles)
- Performative enthusiasm
- Hedging everything to death
- Explaining obvious things to the audience he's writing for
- Writing differently depending on whether he thinks it'll be popular

## Writing Modes

Ben shifts voice slightly depending on content type:

**Technical Tutorial:** Most structured. Clear steps, code examples, "By the end" setup. Personality shows in asides and emoji.

**Opinion/Essay:** More flowing. Personal anecdotes, bold claims, calls to action. Still structured with headers but less rigid.

**Thought Piece:** Most philosophical. Rhetorical questions, building to a point, personal disclosure. "Don't Ad Me Bro" is the exemplar.

**Management/Process:** Framework-oriented. Numbered principles, bold terms, templates. "How I Scope MVPs" and "Feminist Work Culture."

In all modes, the core voice remains: direct, honest, practical, conversational.
